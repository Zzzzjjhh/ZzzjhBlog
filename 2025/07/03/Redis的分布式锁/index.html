这种初级版本存在的问题 如图

详细解释

存在线程 1、线程 2、线程 3 三个线程，具体问题场景如下：

  1. 线程 1 获取锁与业务执行
    线程 1 率先进入，成功获取锁，随后开始执行业务。但线程 1 的业务阻塞时间极长，远超锁的超时时间，导致业务还未执行完，锁已自动释放。
  2. 线程 2 获取锁
    此时锁处于释放状态，线程 2 进入并成功获取到锁，开始执行业务。
  3. 线程 1 释放锁的问题
    线程 1 的业务最终完成，执行释放锁操作。但此时它释放的锁，实则是线程 2 当前持有的锁（原锁因超时已释放，线程 2 重新获取了同名锁）。
  4. 线程 3 获取锁与线程安全隐患
    线程 3 也在此阶段获取到锁。最终可能出现多个线程（如线程 2 和线程 3）并行执行业务的情况，破坏业务原子性，引发严重的线程安全问题。

这种问题误删锁的问题 我们解决方法就是在 释放锁的时候进行一个判断是否是自己持有的锁

改进Redis的分布式锁

为了解决上面的问题我们可以 在存入线程的时候 用UUID表示
为什么不用线程id？
因为分布式集群情况下 线程id是递增的 这个是很容易重复的会导致问题 所以使用uuid

在释放锁的时候 先获取锁中的线程 判断是否和当前线程标识一致
如果一致则释放锁
不一致则不释放锁

这样改进之后 还存在一个问题 就是在极端情况下我们在释放锁的时候阻塞了

为什么释放锁会阻塞？

问题分析
当前释放锁的逻辑（先获取值校验，再删除）是非原子操作。即使添加了线程 ID 校验**，若在 get（获取锁值）和 delete（删除锁）之间发生 JVM 阻塞（如 GC 停顿）**，锁可能因超时而被其他线程重新获取。当原线程恢复后执行 delete，会误删其他线程持有的锁，最终导致线程安全问题。

解决这种问题的方案

使用Lua脚本来保证原子性 就是判断锁是否是自己的 和释放锁 要么一起成功 要么一起失败

为什么 Lua 脚本能保证原子性？

  1. Redis 的单线程模型
    Redis 是单线程执行命令的，当一个 Lua 脚本被提交后，它会被完整执行完毕，不会被其他命令（如其他线程的锁获取请求）打断。因此，脚本中的 GET（校验锁）和 DEL（释放锁）操作是原子化的，不存在时间差。
  2. 脚本逻辑不可分割
    在 Lua 脚本中，判断锁是否属于当前线程（GET）和删除锁（DEL）是同一代码块，要么全部执行，要么全部不执行。即使在脚本执行过程中发生 JVM 阻塞（如 GC），其他线程的锁请求也无法插入，因此不会出现 “误删其他线程锁” 的情况。

基于setnx实现的分布式锁存在下面的问题

不可重入：

同一个线程无法多次获取同一把锁的问题

不可重入可能造成的问题 首先在方法A中调用方法B 我们先要获取锁才能调用方法B 然后获取锁之后 我们在方法B中又要获取同一把锁就会导致死锁的情况

不可重试

获取锁的时候之尝试了一次就返回false 没有重试机制

超时释放

锁的超时释放虽然可以避免死锁 但是如果业务执行耗时较长 也会导致锁的释放 存在安全隐患

主从一致性

主从异步复制导致的锁丢失

  • 问题描述：
    • Redis 主从复制是异步的。当客户端 A 在主节点成功获取锁后，主节点可能尚未将锁数据同步到从节点，此时主节点宕机。
    • 从节点晋升为主节点后，新主节点中没有锁数据，客户端 B 可以再次获取同一把锁，导致锁失效

解决这些问题非常的麻烦 所以我们使用成熟的框架Redission 当然上面的方法 在大部分场景下面都能 使用 因为发生上面4种情况的概率很低
